# 0053 — x86_64 v128 XMM spill: shape-aware allocator + MOVUPS helpers

- **Status**: Accepted
- **Date**: 2026-05-11
- **Author**: zwasm-from-scratch loop (chaploud)
- **Tags**: jit, regalloc, x86_64, simd, v128, spill

## Context

The last remaining named OrbStack simd_assert FAIL post-§9.9-h-7 is
`simd_bitwise.17` (D-078 (c)). The fixture's 12 functions chain
deeply-nested `block (result v128)` constructs whose total live v128
vregs exceed the 6 allocatable XMMs (XMM8..XMM13 per ADR-0041's
SSE4.2 baseline). When the regalloc spills, `resolveXmm` raises
`Error.UnsupportedOp` for `.spill` slot ids:

```
x86_64/gpr: resolveXmm rejected spilled vreg=63 spill_off=16 (handler not XMM-spill-aware)
```

Mac aarch64 PASSES the same fixture because ARM64's `qLoadSpilled` /
`qStoreSpilled` use 16-byte `LDR Q` / `STR Q` against FP-frame slots
that the v128-aware spill layout reserves on ARM64. x86_64 has no
equivalent.

Three structural barriers stack:

1. **Allocator spill stride is uniform 8 bytes.**
   `Allocation.slot()` returns
   `(id - max_reg_slots_gpr) * 8` for spill offsets — every slot is
   8 bytes regardless of vreg shape. A v128 vreg occupying a single
   slot id would only have 8 spill bytes; a 16-byte MOVUPS store
   would overflow into the next slot.

2. **Existing FP spill helpers use 8-byte MOVSD.**
   `gpr.xmmLoadSpilled` / `xmmStoreSpilled` emit `MOVSD`-form
   load/store via `rbpLoadXmmF64Auto` / `rbpStoreXmmF64Auto`. The
   helpers are correct for f32/f64 (the upper 64 bits are dead) but
   silently truncate v128 writes if reused.

3. **~145 `resolveXmm` call sites** in `src/engine/codegen/x86_64/
   op_simd.zig` plus 2 in `op_globals.zig`, 4 in `op_call.zig`, and
   3 in `op_control.zig` reject `.spill` slots outright. Even when
   the helpers exist, every v128 emit handler that touches a v128
   vreg needs to switch from `resolveXmm` to a spill-aware helper.

The same barrier also gates non-bitwise v128 fixtures whose vreg
pressure happens to stay under the XMM cap today; adding handlers
for any future v128 op that creates additional live vregs (e.g. a
6-input pattern) will incrementally re-trigger the same UnsupportedOp.

## Decision

**Land a three-part fix in a dedicated chunk family `§9.9-h-8..-h-N`**:

### Part 1 — Shape-aware allocator spill sizing

Extend `regalloc.Allocation`'s spill computation to consult
`shape_tags[vreg]`:

- Scalar vregs (`ShapeTag.scalar`) keep the existing 8-byte stride.
- v128 vregs (`ShapeTag.v128`) occupy **two consecutive slot ids**
  via post-allocation coalescing, OR the allocator's spill-frame
  layout interprets v128 slots as 16-byte regions and bumps
  subsequent slot ids accordingly. The post-allocation coalescing
  variant is preferred — it keeps the allocator's slot-id
  assignment class-blind (per ADR-0018 §"option (b)") and pushes
  shape interpretation into the frame-layout helper.

ARM64's `qStoreSpilled` already requires 16-byte alignment via the
Q-form LDR/STR's imm12-scaled-by-16 encoding; the parallel x86_64
helpers will require it too even though MOVUPS itself is unaligned
(alignment matters for the *allocator's* spill-region layout, not
for the instruction).

### Part 2 — MOVUPS-based v128 spill helpers

Add three helpers in `src/engine/codegen/x86_64/gpr.zig` mirroring
the GPR/XMM spill API:

```zig
pub fn xmmLoadSpilledV128(allocator, buf, alloc, spill_base_off, vreg, stage_idx) Error!Xmm
pub fn xmmDefSpilledV128(alloc, vreg, stage_idx) Error!Xmm
pub fn xmmStoreSpilledV128(allocator, buf, alloc, spill_base_off, vreg, stage_idx) Error!void
```

Use `MOVUPS xmm, [RBP + disp]` / `MOVUPS [RBP + disp], xmm`
(16-byte unaligned, single instruction). The stage register pool
remains `abi.fp_spill_stage_xmms` (XMM14/XMM15); stage_idx 0/1
distinguishes the two operands of a binary op so they don't collide.

The new helpers are **distinct from** `xmmLoadSpilled` /
`xmmStoreSpilled` (which keep MOVSD semantics for scalar FP) — call
sites pick by shape, not by helper.

### Part 3 — Handler migration

Migrate the ~145 `resolveXmm` call sites in `op_simd.zig` to the
new helpers when the vreg is v128-class. Per-handler shape:

- For an op like `v128.and lhs, rhs → result`, replace:
  - `resolveXmm(lhs_v)` → `xmmLoadSpilledV128(..., lhs_v, 0)`
  - `resolveXmm(rhs_v)` → `xmmLoadSpilledV128(..., rhs_v, 1)`
  - `resolveXmm(result_v)` → `xmmDefSpilledV128(..., result_v, 0)`
  - Add `xmmStoreSpilledV128(..., result_v, 0)` after the body.

The migration is mechanical — same shape as the GPR-side §9.7
spill migration (D-045 chunks 13-14). Test sites in
`op_simd_test.zig` need new alias / spill tests parallel to the
existing GPR-spill tests.

### Co-deliverable: `op_simd.zig` source split (D-057)

`src/engine/codegen/x86_64/op_simd.zig` is at 4554 LOC (§A2 hard cap
2000). The handler migration will add roughly 200 LOC for the
spill-aware wrappers, pushing the breach further. Split the file per
op family as a co-deliverable in the same chunk family:

- `op_simd_int.zig` — int arith / cmp / bitwise / lane
- `op_simd_fp.zig` — FP arith / cmp / convert
- `op_simd_mem.zig` — load / store / splat / load_lane / store_lane
- `op_simd_shuffle.zig` — swizzle / shuffle / extract / replace
- `op_simd.zig` retained as a re-export façade / dispatch only

This matches D-057's intended structure and aligns the ARM64
`inst_neon.zig` + `op_simd.zig` source-split (D-065) in shape.

## Alternatives considered

### Alternative A — `resolveXmm` becomes spill-aware uniformly

- **Sketch**: extend `resolveXmm`'s signature to take allocator + buf
  + spill_base_off + stage_idx, falling through to a MOVUPS load
  when the slot is `.spill`. Every existing call site updated.
- **Why rejected**: 145+ sites would all need API change in one go.
  No incremental migration possible. The handler-shape change (need
  paired `xmmStoreSpilledV128` after the body) cannot be captured
  by a "resolve" function alone — every binary-op handler needs to
  know whether to store back. So the API change doesn't actually
  reduce the migration scope, only complicates it.

### Alternative B — Allocator gives v128 vregs 16-byte slot stride

- **Sketch**: change `Allocation.slot()` to return
  `id * 16` for v128-shaped vregs; spill-frame layout doubles in
  size; `xmmLoadSpilled` / `xmmStoreSpilled` switch to MOVUPS
  uniformly (also for f32/f64 — they pay the 2× space cost).
- **Why rejected**: doubles f32/f64 spill memory unnecessarily.
  Forces every FP handler to compute disp via the v128 stride even
  when the vreg is scalar. Cleaner shape is shape-tag-aware spill
  layout (Part 1) keeping scalar FP at 8-byte stride.

### Alternative C — Tail-grow XMM pool

- **Sketch**: add 2 more allocatable XMMs (XMM6..XMM7 → previously
  reserved as stash scratch per D-066). Pushes spill threshold
  from 6 to 8 v128 vregs. Avoids the spill question for
  simd_bitwise.17.
- **Why rejected**: D-066 / §9.9-h-5 reservations are load-bearing
  for the alias-stash pattern. Reclaiming XMM7 forfeits the
  preventive alias guard for FP replace_lane handlers. Even
  reclaimed, 8 XMMs is not enough — simd_bitwise.17's 12 funcs
  have peak live counts > 8. The fundamental scaling fix is
  spill; growing the pool is a delayed bandage.

### Alternative D — Defer to Phase 10+ / mark fixture as SKIP

- **Sketch**: SKIP simd_bitwise.17 via an ADR; close §9.9 with the
  one known OrbStack FAIL absorbed; defer the spill work to Phase
  10's structural cycle alongside D-057 source-split.
- **Why rejected**: §9.9 exit criterion is `fail = skip = 0 on the
  3-host gate`. SKIP-ing trades one form of non-zero for another;
  the row stays `[ ]` either way. The work is structurally bounded
  by Parts 1-3 above; deferral is procrastination, not de-scoping.

## Consequences

### Positive

- **Closes D-078 (c)** — the last named OrbStack visible FAIL.
- **Aligns x86_64 with ARM64 design parity** — ARM64's
  `qLoadSpilled` already takes the 16-byte stride; this brings
  x86_64 to feature parity.
- **Unblocks higher-vreg-pressure v128 ops** — future v128 op
  additions (e.g. wide-input patterns, AVX-extended ops post-Phase
  15) inherit the spill-safe shape from day 1.
- **Forces the D-057 source-split** — co-deliverable lands the
  op_simd.zig source-split that has been pending since §9.7.
  Splits will reduce mid-chunk LOC growth ratchet and improve
  reviewability.

### Negative

- **Multi-chunk effort** — Parts 1-3 + co-deliverable D-057 split
  is realistically 4-6 chunks of bounded scope each. Not a single
  session deliverable. Phase 9 close depends on it.
- **Spill-frame size grows** — modules with many v128 vregs pay
  16 bytes per spilled v128 (was effectively 8 bytes per scalar
  slot times 2 for v128). Acceptable per the spec testsuite's
  module sizes; tighter accounting is Phase 15+ optimisation
  territory.
- **Allocator class-awareness creep** — Part 1 nudges the regalloc
  toward shape-aware accounting. ADR-0018's "class-aware boundaries"
  amendment landed in §9.7 for register pool sizing; this extends
  to spill sizing. Still no full class-aware allocator (D-036
  §"option (b)") — that remains a Phase 15 candidate.

### Neutral / follow-ups

- **D-078 (c) row body update**: at this ADR's first
  implementation chunk landing, narrow the debt row to "Part 1
  pending" / "Part 2 pending" / etc. as each Part lands.
- **op_simd_test.zig parallel split**: when D-057 splits
  op_simd.zig, the test file (2624 LOC, also breaches §A2 cap)
  splits the same way.
- **D-066 alias-stash audit** — preventive grep of
  `encMovapsXmmXmm(dst,` sites (handover Next-candidate carry-over)
  remains a separate preventive audit. Independent of this ADR.
