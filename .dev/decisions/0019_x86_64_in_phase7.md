# 0019 — Pull x86_64 backend into Phase 7 (operationalise P7 backend equality)

- **Status**: Accepted
- **Date**: 2026-05-04
- **Author**: Shota / autonomous loop
- **Tags**: roadmap, phase7, jit, x86_64, multi-arch

## Context

ROADMAP §2 P7 declares: "Both backends are equal — any feature
that exists in `jit_arm64/` exists in `jit_x86/`. No 'ARM64-only'
or 'x86-only' implementations."

But the original §9 phase plan staggers the backends:

- **Phase 7**: Single-pass JIT v1 (ARM64 only).
- **Phase 8**: Single-pass JIT v1 (x86_64).

This staging directly contradicts P7. Specifically:

1. **Three-host gate is asymmetric**. Mac aarch64 runs JIT;
   OrbStack Ubuntu x86_64 + windowsmini both `error.SkipZigTest`
   on JIT-touching tests. The §9.7 / 7.4d "spec testsuite via
   JIT" gate that was about to land would have been Mac-only —
   meaning Linux + Windows would have shipped pages of JIT code
   for an entire Phase without ever running them.
2. **Design drift accumulates undetected**. Caller-supplied
   skeleton (X24..X28 invariants, ADR-0017's predecessor),
   regalloc pool, ABI tables — all designed against AAPCS64 only.
   Bringing in x86_64 in Phase 8 would surface every System V vs.
   AAPCS64 mismatch as Phase-8 work, with the original designer
   no longer having the design rationale loaded.
3. **W54-class regression risk**. v1's hardest bugs were caused
   by post-hoc layered changes. Adding x86_64 on top of an
   ARM64-shaped substrate is exactly that pattern.
4. **CI value undelivered**. The `windowsmini` SSH host + OrbStack
   VM exist precisely to catch x86_64-specific bugs early. With
   the JIT skipped on those, they're vestigial during Phase 7.

The §9.7 / 7.4 spec gate work just exposed the asymmetry
directly: sub-7.4a/b/c shipped JIT runtime infra (jit_mem,
linker, entry frame) on Mac aarch64, with Linux + Windows tests
returning `error.SkipZigTest`. Continuing this trajectory would
land 7.4d/7.5/7.6/7.7/7.8 (multiple cycles of work) all skipped
on 2 of the 3 hosts.

## Decision

**Reshape Phase 7 to develop ARM64 and x86_64 in parallel within
the same phase, sharing common infrastructure under `src/jit/`.**

### Phase 7 task table (post-ADR-0019)

The existing §9.7 has rows 7.0–7.8. The reshape:

```
7.0  reg_class                                                    [x] EXISTING (already closed)
7.1  regalloc                                                     [x] EXISTING — implementation cycle re-touches per ADR-0018
7.2  ARM64 inst/abi tables                                        [x] EXISTING — implementation cycle re-touches per ADR-0017
7.3  ARM64 emit (ZIR → bytes)                                    [ ] EXISTING — implementation cycle re-touches per ADR-0017 prologue
7.4  JIT runtime infra (jit_mem / linker / entry)                 [ ] **NEW ROW** — extracts the sub-7.4a/b/c work landed under
                                                                       what was the old 7.4 spec-test row
7.5  spec testsuite via ARM64 JIT (pass=fail=skip=0)              [ ] EXISTING content (was 7.4) — moved one slot down
7.6  x86_64 reg_class + abi tables                                [ ] **NEW ROW**
7.7  x86_64 emit (ZIR → bytes)                                    [ ] **NEW ROW**
7.8  spec testsuite via x86_64 JIT (pass=fail=skip=0)             [ ] **NEW ROW**
7.9  realworld via ARM64 JIT (40+ samples)                        [ ] EXISTING content (was 7.5)
7.10 realworld via x86_64 JIT                                     [ ] **NEW ROW**
7.11 three-way differential (interp == jit_arm64 == jit_x86)      [ ] EXISTING was 2-way (7.6); **scope expanded to 3-way**
```

Rows already `[x]` (7.0/7.1/7.2) stay `[x]` per ROADMAP §18.3
("edit in place as if it had always been so"). The implementation
cycles for ADR-0017 and ADR-0018 will re-touch files those rows
produced (e.g. `regalloc.zig`, `abi.zig`); git history shows the
journey, the row's closure stands. New rows 7.4/7.6/7.7/7.8/7.10
are inserted; existing-content rows 7.5/7.9/7.11/7.12 shift down
or get scope-expanded as marked.

### Phase 8 redefined

Existing ROADMAP §9.8 declares Phase 8 as "JIT v1 x86_64
baseline 🔒" with the locked exit criterion of "x86_64 emit +
spec gate + realworld + three-way differential". Since Phase 7
now covers both backends per this ADR, that work transfers into
Phase 7's expanded task table (§9.7 / 7.6–7.11 above).

**Phase 8 is redefined as JIT optimisation foundation**
(post-Phase-7-baseline):

- hoist / coalesce / live-range splitting (the W43 / W44 / W45
  v1 work, ported onto v2 ZIR substrate per P14)
- better regalloc heuristics (linear-scan with reuse, vs today's
  greedy-local)
- AOT skeleton (was Phase 12; partial transfer)

The 🔒 lock attached to Phase 8 (originally x86_64) carries
**into Phase 7.11** (three-way differential) — the cross-arch
parity gate is preserved, just lands one Phase earlier.

Phase 9+ phase scopes unchanged; numbering stays. Phase 12 (AOT)
loses some scope to Phase 8 but retains AOT-specific finalisation.

### Common infrastructure boundary

`src/jit/` (Zone 2, shared) holds:

- `regalloc.zig` — slot-id allocator, arch-neutral
- `linker.zig` — JitModule, fixup patching (per-arch dispatch
  function for the actual byte-edit step)
- `entry.zig` — trampoline JIT-emitter (per-arch),
  Zig-callable shim that sets up `*const JitRuntime` arg
- `reg_class.zig` — semantic categories (gpr / fp / inst_ptr /
  spill), arch-neutral

`src/jit_arm64/` and `src/jit_x86/` hold:

- `inst.zig` — per-arch encoders + tests
- `abi.zig` — per-arch ABI tables (`reserved_invariant_gprs`,
  `allocatable_gprs`, `arg_gprs`, etc.)
- `emit.zig` — per-arch ZIR→native emit pass

A3 (cross-arch import ban) stays enforced; both per-arch
backends import from `src/jit/` only.

### Three-host gate after ADR-0019

| Host | Backend run | Status |
|------|-------------|--------|
| Mac aarch64 | interp + ARM64 JIT | full |
| OrbStack Ubuntu x86_64 | interp + x86_64 JIT | full |
| windowsmini x86_64 | interp + x86_64 JIT | full |

Every host runs interp + its native JIT. Differential check
asserts `interp == native_jit`. Phase 7.11 adds the cross-arch
check (Mac runs both via QEMU? cross-compile? — design in 7.11
itself).

## Alternatives considered

### Alternative A — Keep original ARM64→Phase 8 staging

- **Sketch**: Continue with ARM64-only Phase 7. Plan x86_64 as
  Phase 8 dedicated.
- **Why rejected**: directly contradicts P7. Three-host gate
  asymmetry. Design drift risk. Already painful at sub-7.4c
  (JIT tests skip on 2/3 hosts).

### Alternative B — Phase 7 = ARM64 foundation, Phase 8 = x86_64 + ARM64 polish

- **Sketch**: Phase 7 ships an ARM64-only "foundation" (regalloc,
  emit, basic spec gate). Phase 8 adds x86_64 + retroactively
  fixes any ARM64-specific assumptions surfaced by the 2nd
  backend.
- **Why rejected**: "retroactively fixes" is exactly the
  W54-class anti-pattern. Better to surface mismatches as Phase
  7 cycles them through.

### Alternative C — Phase 7 = both arches, but x86_64 stops at "compile + verify, no execute"

- **Sketch**: x86_64 backend emits bytes + runs through a
  validator, but doesn't execute (no spec gate against it).
- **Why rejected**: validator-only verification doesn't catch
  ABI mismatches; only execution does.

### Alternative D — Add P15 "Multi-arch parity by construction"

- **Sketch**: Promote the parity principle to a numbered
  inviolable principle.
- **Why rejected**: P7 already says it ("Both backends are
  equal"). The deviation is the **operationalisation** (phase
  staging), not the principle. Adding P15 would be redundant.

## Consequences

### Positive

- **P7 operationalised**, not just declared. Phase 7 ends with
  feature-complete x86_64 alongside ARM64.
- **3-host gate restored to symmetry**. Linux + Windows fully
  exercise the JIT.
- **Design pressure surfaces ABI mismatches early** in the same
  Phase the abstractions are designed.
- **Phase 8 elevation**: instead of "x86_64 catch-up", Phase 8
  becomes "optimisation foundation" — more valuable, more
  motivating, better placed in the timeline.

### Negative

- **Phase 7 timeline doubles** vs. ARM64-only. Specifically:
  7.6/7.7/7.8/7.10 are new sub-rows. Order-of-magnitude
  estimate: previously 5-7 cycles remaining; now 12-18.
- **Two emit.zig files to maintain**. Each per-arch emit will
  add ~3000-4000 lines mirroring the other.
- **Differential test infra is heavier**: 3-way (interp ==
  arm64_jit == x86_64_jit) is more state than 2-way.
- **Design must abstract earlier**: every per-arch decision
  forces a shared-vs-per-arch placement choice in real time
  (rather than retrofitting later).

### Neutral / follow-ups

- **Cross-arch differential**: Mac aarch64 host can't run x86_64
  bytes natively. Options for 7.11:
  - cross-compile spec runner for x86_64 + run on OrbStack
    against the same fixtures
  - QEMU on Mac (heavy)
  - shared assertion log: each host emits "this fixture produced
    these outputs", a meta-runner diffs.
  Decision deferred to 7.11 itself.
- **AOT (was Phase 12)** partially folds into Phase 8 (since
  Phase 8 is now optimisation + AOT skeleton).
- **A8 (Windows verified pre-v0.1.0)** in §2.1 stays as-is;
  ADR-0019 makes Windows JIT exercise from Phase 7 onward
  (stronger than A8's "pre-v0.1.0").
